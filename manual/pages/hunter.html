🎯 HUNTER — Búsqueda cross-source

HUNTER es el cazador de licitaciones de Licitómetro. Cuando encontrás un proceso en una fuente (por ejemplo en el Boletín Oficial), HUNTER sale a buscarlo en todas las demás (COMPR.AR, ComprasApps, municipios) y fusiona los datos faltantes automáticamente.

¿Por qué HUNTER existe?

Muchas licitaciones se publican en más de una fuente. El Boletín Oficial anuncia el decreto, COMPR.AR Mendoza publica el pliego electrónico con los ítems, y ComprasApps guarda el histórico. Cada fuente tiene datos distintos: el Boletín suele tener encuadre legal y expediente, COMPR.AR tiene ítems estructurados y cronograma, ComprasApps tiene CUCs y jurisdicción.

Antes de HUNTER, cada versión del mismo proceso quedaba como una licitación separada. Ahora, con un click en "Enriquecer datos", HUNTER:

  1. Extrae identificadores del texto (decreto, expediente, número de licitación, resolución, CD)
  2. Busca esos identificadores en todas las otras licitaciones de la base
  3. Si no encuentra por número, hace fallback a búsqueda por keywords del título (MongoDB $text index con score >= 3.0)
  4. Por cada match, mergea los datos faltantes (nunca sobrescribe datos existentes)

Cómo usarlo

En la vista de detalle de cualquier licitación, hacé click en el botón 🔄 Enriquecer de los botones flotantes (admin). Esto dispara el pipeline de 4 fases:

  1. Enrichment por fuente — descarga HTML/PDF y extrae datos (label-based para COMPR.AR, CSS selectors para HTML, pypdf para PDFs)
  2. HUNTER cross-source — busca identificadores + fallback por title keywords
  3. Nodo re-matching — re-evalúa todos los nodos activos
  4. Response — devuelve campos actualizados + resumen de HUNTER
⚠️ Bug resuelto (Mar 22, 2026)

Antes, las licitaciones de COMPR.AR hacían return al final de la fase 1 y se saltaban HUNTER + nodo matching. Fix: restructurado en 4 fases secuenciales que siempre corren para TODAS las fuentes.

Identificadores reconocidos

HUNTER busca 5 patrones regex en el texto de la licitación (título + objeto + description):

TipoEjemplos
DecretoDecreto 140/2026, Dec. N° 140/26
ExpedienteExpte 123456/2024, EX-2026-01172061-GDEMZA-CCC
LicitaciónLicitación Pública 45/2026, LP 5/26
ResoluciónResolución 100/2024, Res. N° 50/26
Contratación DirectaCD 20/2026, contratación directa nº 15/26

Normalización de años

Los años de 2 dígitos se normalizan: 24-27 → 2024-2027. Años >= 28 se rechazan (imposibles en el futuro cercano). Años < 24 se marcan como archivados.

Título keyword fallback

Si HUNTER no encuentra números de identificación, hace búsqueda por palabras clave del título usando MongoDB $text:

  • Filtra stopwords españolas (de, del, la, en, y, para, con, por...)
  • Requiere palabras de 4+ caracteres
  • Necesita 3+ palabras significativas para activarse
  • Usa las top 5 palabras como query AND
  • Solo acepta matches con textScore >= 3.0 (filtro de ruido)
ℹ️ Ejemplo

Título: "Adquisición de calefactores eléctricos para escuelas rurales" → palabras significativas: adquisicion, calefactores, electricos, escuelas, rurales. HUNTER busca esas 5 palabras AND en el índice de texto y devuelve solo matches con score >= 3.0.

Merge de datos (idempotente)

Cuando HUNTER encuentra un match, llama a merge_source_data() que:

  • Solo llena campos que están vacíos o nulos en el destino
  • Nunca sobrescribe datos existentes
  • Registra el merge en metadata.cross_source_merges[] (últimos 10)
  • Actualiza proceso_id canónico si cambia

Esto significa que podés correr "Enriquecer" N veces sin duplicar ni corromper datos. Es completamente idempotente.

Response del endpoint

El botón Enriquecer llama POST /api/licitaciones/{id}/enrich que devuelve:

{
  "success": true,
  "fields_updated": 5,
  "fields": ["description", "objeto", "budget", "opening_date", "expedient_number"],
  "hunter": {
    "matches_found": 1,
    "merged_from": [
      {
        "id": "68e1a2...",
        "fuente": "Boletin Oficial",
        "title": "Decreto 140/2026"
      }
    ],
    "fields_merged": ["budget", "opening_date", "expedient_number"]
  }
}

Panel HUNTER en el frontend

En el detalle de la licitación, el botón 🎯 Hunter abre un panel lateral que muestra:

  • Estado: cargando / N matches encontrados / sin matches
  • Cards de cada match: fuente, título, score, similitud, campos que podés mergear
  • Botón "Fusionar" por match individual (override manual del merge automático)
  • Estadísticas: estrategias usadas (identifier, title-keyword, both)
  • Si no hay matches: botón "Buscar más profundo" que lanza una búsqueda web (Firecrawl) para encontrar el proceso en Google

Comparar precios

Cuando HUNTER encuentra el mismo proceso en ComprasApps (histórico de adjudicaciones), podés ver precios reales de compras pasadas. Esto es la base del motor de Inteligencia de Precios de CotizAR: te permite cotizar sabiendo qué pagó el Estado antes por ítems similares.

Infraestructura

backend/utils/proceso_id.pyextract_identifiers_from_text() (5 regex), normalize_proceso_id()

Además corre automáticamente después de cada scraping (scheduler_service.auto_link_after_scrape()) pero solo cuando el scraper provee un proceso_id. El enriquecimiento manual siempre corre HUNTER, hay o no hay proceso_id.

💡 Tip de uso

Si una licitación parece "incompleta" (sin presupuesto, sin apertura, sin expediente), siempre probá HUNTER primero. La mayoría de las veces esos datos están en otra fuente y HUNTER los fusiona automáticamente.
