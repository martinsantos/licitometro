🧪 Lab experimental Admin

El Lab (/lab) es el área de pruebas para scrapers, extractores de PDF y parsers de HTML. Podés probar nuevas fuentes sin afectar producción, comparar estrategias y validar resultados antes de pasar a admin.

🖥 Lab con Quick Test, Compare scrapers y PDF Parser A/B.

Herramientas disponibles

⚡ Quick Test (Firecrawl)

Ingresás una URL → Firecrawl renderiza el sitio con browser headless (maneja JS) → extrae markdown estructurado. Ideal para:

  • Ver si un sitio es scrapeable antes de crear un scraper
  • Debuggear por qué un scraper devuelve vacío
  • Copiar CSS selectors viendo el HTML procesado
  • Probar sitios JS-heavy (Single Page Apps)

Config: requiere FIRECRAWL_API_KEY en el backend. Si no está configurado, el botón queda deshabilitado.

🔄 Compare scrapers

Corre 2 o 3 scrapers distintos contra la misma URL y compara side-by-side:

ScraperItemsDuraciónSuccess rate

selenium4734.5s100%

Te ayuda a elegir la estrategia más rápida que extrae todos los items.

🤖 Extract con IA

Pegás un HTML o un PDF largo → Groq extrae los campos que le pidas (JSON schema):

{
  "title": "string",
  "expediente": "string",
  "budget": "number|null",
  "opening_date": "date|null",
  "items": [{ "descripcion": "string", "cantidad": "number" }]
}

Útil cuando un sitio no tiene estructura clara y los CSS selectors son inviables.

📄 PDF Parser A/B

Compara dos extractores de PDF lado a lado para el mismo archivo:

LibreríaFortalezaDebilidad
pypdfRápido, output plano, preserva todas las líneasSin estructura jerárquica
opendataloader-pdfOutput estructurado con kids[] (headings, tables, paragraphs)Lento (Java), pierde algunas líneas en PDFs escaneados

La UI muestra los dos outputs side-by-side con diff highlighting. Útil para decidir qué estrategia usar por fuente.

⚠️ Feature flag

OpenDataLoader está detrás de USE_OPENDATALOADER_PDF=true en el backend. Por default está en false porque es más lento y ocasionalmente pierde líneas. Para BOE (Boletín Oficial), el código fuerza pypdf siempre porque la segmentación depende de líneas que opendataloader a veces elimina.

🔗 URL resolver test

Para debuggear sitios con redirecciones complejas (ASP.NET session-state, Cloudflare challenges). Ingresás una URL y el lab muestra la cadena completa de redirects hasta llegar al HTML final, con cookies y headers de cada hop.

📊 Enrichment dry-run

Seleccionás una licitación existente y el Lab corre el pipeline de enriquecimiento en modo dry-run: calcula qué campos cambiarían sin escribir a MongoDB. Útil para validar antes de correr enrichment masivo.

Workflow típico para agregar una fuente

  1. Quick Test la URL del listado → verificar que se renderiza bien
  2. Compare scrapers → identificar la estrategia óptima
  3. PDF Parser A/B si hay pliegos adjuntos → decidir pypdf vs opendataloader
  4. Extract con IA sobre 1-2 items → validar calidad de extracción
  5. Con todo claro → ir a Admin → Importar fuente con los selectors correctos
  6. Activar el scraper y monitorear la primera corrida en Admin → Logs

Historial de experimentos

El Lab guarda los últimos 50 experimentos ejecutados con sus inputs y resultados. Podés volver a correr uno viejo con un click, o duplicarlo para modificar y comparar.

💡 No afecta producción

Todo lo que hacés en el Lab es read-only. Ninguna operación del Lab escribe a la colección licitaciones ni dispara notificaciones. Es un sandbox seguro.
