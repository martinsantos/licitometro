🔧 Panel de Administración Admin

El panel de admin (/admin) es el centro de control de Licitómetro. Desde aquí monitoreás los scrapers, gestionás fuentes, revisás calidad de datos y operás el sistema.

⚠️ Solo admins

Todas las acciones de esta sección requieren rol admin. Si no ves el link "Admin" en el header, no tenés permisos. Los endpoints están protegidos server-side, no solo el UI.

🖥 Vista general del panel de admin con los 10 tabs.

Tabs del panel

El panel tiene 10 tabs agrupados por función:

📊 Monitor

🖥 Monitor: estado de scrapers, errores, cola de enrichment.

Vista general del estado del sistema:

  • Scrapers activos: cuántos corrieron hoy, cuántos fallaron, tiempo promedio
  • Licitaciones por día: gráfico de barras últimos 7 días
  • Errores recientes: últimos 20 errores con stack trace
  • Cola de enriquecimiento: items pendientes de enrich level 2
  • Health checks: MongoDB, nginx, postfix, APScheduler

🌐 Fuentes

🖥 Fuentes: 24 scrapers configurados con última ejecución.

Lista de las 24 fuentes configuradas. Por cada fuente:

  • Nombre + URL base + scraper asignado
  • Última ejecución: timestamp + items nuevos + duración
  • Estado: ✅ OK / ⚠️ Warnings / ❌ Error
  • Botón "Ejecutar ahora" — dispara scraping manual
  • Botón "Ver logs" — últimas 10 corridas con output completo
  • Switch activo/inactivo — pausa sin borrar config

➕ Importar nueva fuente

Wizard para agregar una nueva fuente de datos sin tocar código:

  1. URL del listado: ej. https://ipvmendoza.gov.ar/licitaciones/
  2. Tipo de scraper: generic_html (90% de los casos), o dedicado (COMPR.AR, PDF, etc.)
  3. CSS selectors: el wizard intenta detectar automáticamente, pero podés refinar:
    • item_selector — cómo encontrar cada licitación (ej: article.post)
    • title_selector — dentro del item, dónde está el título
    • date_selector — dónde está la fecha
    • link_selector — cómo extraer el link al detalle
  4. Test en vivo: el wizard scrapea 1 página y te muestra 3-5 items de preview
  5. Guardar: crea el ScraperConfig en MongoDB y lo activa
💡 Tip: usar Lab primero

Antes de importar una fuente nueva, probá en Lab con Firecrawl o OpenDataLoader para ver si el sitio es scrapeable. Te ahorra horas de configuración.

📜 Logs

🖥 Logs: historial de ejecuciones con filtros y stack traces.

Historial completo de ejecuciones de scrapers:

  • Filtros: fuente, estado, rango de fechas
  • Vista de cada run: items procesados, duplicados, nuevos, errores, duración
  • Output del scraper: stdout/stderr capturado
  • Trace de excepciones con stack completo

Botones de acción rápida:

  • Re-clasificar categorías: corre CategoryClassifier sobre items sin categoría
  • Backfill objeto: corre extract_objeto() sobre items sin objeto
  • Re-match nodos: re-evalúa todos los nodos contra todas las licitaciones
  • Deduplicar: busca duplicados por content hash y mergea

💾 Almacenamiento

Uso de disco por categoría:

  • PDFs descargados (pliegos)
  • Cache de HTTP (pliego_url_cache.json)
  • Logs de scrapers
  • Backups de MongoDB

Botón "Cleanup" corre storage_cleanup_service que elimina PDFs más viejos que STORAGE_MAX_MB (default 500MB), rota logs con > LOG_RETENTION_DAYS (default 7 días) y limpia cache expirado.

⚙️ Configuración de scrapers

Editor inline de los ScraperConfig en MongoDB. Editá CSS selectors, frecuencia, timeouts, headers custom — todo sin redeploy. Los cambios se cargan en el próximo run del scheduler.

⚠️ Scheduler load-at-startup

El scheduler_service carga las configs al inicio del backend. Si editás una config crítica (ej: cambiás la URL base), puede que necesites reiniciar el backend para que el scheduler vea el cambio. En cambio, el matcher de nodos recompila regex al guardar — esos cambios son inmediatos.

🗂️ Gestión de licitaciones

CRUD directo sobre la colección:

  • Buscar por título, expediente, organización
  • Editar campos manualmente (útil para corregir fechas malas)
  • Cambiar workflow_state (descubierta → evaluando → ... → presentada)
  • Eliminar duplicados manualmente
  • Ver historial de cambios (metadata.estado_history)
  • Re-enriquecer con botón "🔄 Enriquecer"
  • Publicar con link público (genera slug en /p/:slug)

🔬 Sistema

Health, cache, backups:

  • Health: ping a MongoDB, test de SMTP, Telegram API reachable
  • Cache: ver tamaño, invalidar, limpiar
  • Backups: listar backups disponibles (/opt/licitometro/backups/), trigger manual, restore
  • Variables de entorno: ver las env vars activas (passwords redacted)
  • Reiniciar servicios: backend, scheduler, nginx (requiere sudo)

Workflow de estados (manual)

El estado de una licitación es business-critical. Nunca se cambia automáticamente — siempre vos, manualmente, desde el panel o desde el stepper del detalle.

EstadoSignificado
descubiertaRecién scrapeada, sin revisar
evaluandoEstás analizando si cotizar
preparandoActivamente armando la oferta
presentadaOferta entregada
descartadaDecidiste no presentarte
🚨 Incidente histórico (Feb 12, 2026)

Hubo un bug en enrichment_cron_service que auto-transicionaba TODAS las licitaciones de descubierta a evaluando cada 30 minutos. 1,349 items se corrompieron. Se revirtió manualmente y la regla quedó grabada en piedra: enrichment no toca workflow_state, jamás.

Ejecutar scripts manualmente

Para correr scripts de mantenimiento en producción:

ssh root@76.13.234.213 \
  "docker exec -w /app -e PYTHONPATH=/app licitometro-backend-1 \
   python3 scripts/NOMBRE_SCRIPT.py"

El flag -w /app -e PYTHONPATH=/app es crítico — sin eso, los imports de services.* fallan.

Backup & restore

Los backups automáticos viven en /opt/licitometro/backups/mongodb_YYYYMMDD_HHMMSS.gz. Retención: 7 días.

# Backup manual
ssh root@VPS "bash /opt/licitometro/scripts/backup-mongodb.sh"

# Listar backups
ssh root@VPS "ls -lh /opt/licitometro/backups/"

# Restore (requiere confirmación "yes")
ssh root@VPS "bash /opt/licitometro/scripts/restore-mongodb.sh \
  /opt/licitometro/backups/mongodb_20260408_120000.gz"
🚨 NUNCA usar docker compose down -v

Ese comando elimina los volumes de MongoDB → pérdida total de datos. Siempre usar docker restart o el script scripts/deploy-prod.sh que hace backup automático pre-deploy.
